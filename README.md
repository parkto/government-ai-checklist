대한민국 정부 AI 응용 서비스 개발 매뉴얼 (Draft)
1.	매뉴얼의 목적과 기본 원칙
이 매뉴얼은 “AI를 쓰는 것”이 아니라 “문제를 해결하는 공공 서비스”를 만들기 위한 실무 지침이다.
AI는 수단이며, 모든 단계에서 다음 원칙을 따른다.
●	문제 중심 (Problem-first)
●	사용자 중심 (User-first)
●	점진적 검증 (Test early, fail fast)
●	책임성과 설명 가능성
●	지속 가능성과 모델 독립성
 
1.	문제 정의
무슨 문제를 풀려고 하는가?
1-1. 문제 서술의 기준
●	기술이 아닌 행정·시민의 실제 불편을 기준으로 정의
●	“AI로 무엇을 할 것인가”가 아니라 “무엇이 불편한가”
●	현재 제도·업무 구조 하에서 반복·지연·오류가 발생하는 지점 명시
1-2. 문제 정의 체크리스트
●	누가 이 문제로 고통받는가? (시민 / 공무원 / 둘 다)
●	얼마나 자주 발생하는가?
●	해결하지 않을 경우의 사회적·행정적 비용은 무엇인가?
●	법·제도 변경 없이 기술로 개선 가능한 영역인가?
1-3. AI 적용 적합성 판단
●	반복적 판단이 많은가?
●	대량의 문서·데이터를 다루는가?
●	사람의 전문성이 병목이 되는가?
●	완전 자동화가 아닌 “보조”로도 효과가 있는가?
 
2.	데이터 정의
문제를 풀기 위해 필요한 데이터는 무엇인가?
2-1. 데이터 리스트업
●	업무 데이터 (행정시스템, 업무 이력, 민원 데이터)
●	법·규정·지침 문서
●	외부 데이터 (타 부처, 공공데이터, 통계)
●	실시간/준실시간 데이터 필요 여부
2-2. 데이터 속성 명시
각 데이터에 대해 다음을 반드시 명시한다.
●	소유 부처
●	생성 주기
●	최신성 요구 수준
●	민감도 (개인정보, 민감정보, 비식별 가능 여부)
●	정형/비정형 여부
2-3. AI 관점 데이터 점검
●	학습용/추론용 구분
●	최신성 유지 필요 여부
●	규칙 기반으로 대체 가능한 영역 여부
●	데이터 편향 가능성
 
3.	부처 간 협업 방안
누가 함께 해야 하는가? 어떻게 협업할 것인가?
3-1. 협업 부처 식별
●	데이터 보유 부처
●	제도·정책 소관 부처
●	실제 업무 수행 부처
●	플랫폼·인프라 운영 부처
3-2. 협업 구조
●	주관 부처 1곳 명확화
●	데이터 제공 부처는 “참여자”가 아니라 “공동 설계자”
●	AI 모델·시스템은 중앙 공통 자산으로 설계
3-3. 협업 방식
●	초기 설계 단계부터 공동 워크숍
●	데이터·법률·현업 담당자 동시 참여
●	정기적 리뷰(월/분기 단위)
 
4.	As-Is / To-Be 정의
현재는 어떠하며, 결과 이미지는 무엇인가?
4-1. As-Is
●	현재 업무 흐름 다이어그램
●	소요 시간, 병목 지점
●	사람 의존 구간
4-2. To-Be
●	AI 도입 후 업무 흐름
●	자동화/보조/판단 분리
●	사람의 역할 재정의
4-3. 변화의 범위 명시
●	업무 단축 시간
●	판단 품질 개선 여부
●	시민 체감 변화
 
5.	고객(사용자) 조사
이 문제가 실제 문제임을 어떻게 확인하는가?
5-1. 사전 검증
●	인터뷰 (시민, 공무원)
●	민원·콜센터 로그 분석
●	실제 업무 관찰(Shadowing)
5-2. 사용자 만족도 측정
●	서비스 이용 후 간단 설문
●	업무 시간 절감 체감 여부
●	신뢰도 및 재사용 의사
5-3. 사용률 기반 검증
●	실제 사용 빈도
●	반복 사용 여부
●	AI 추천/응답 채택률
 
6.	서비스 운영 지표 (KPI)
무엇을 반드시 측정해야 하는가?
6-1. 필수 지표
●	사용자 수 (DAU/MAU)
●	처리 시간 단축률
●	자동화/보조 성공률
●	오류·이탈률
6-2. AI 특화 지표
●	모델 응답 정확도
●	인간 검토 개입 비율
●	예외 처리 비율
●	데이터 최신성 지표
6-3. 자동 수집 원칙
●	로그 기반 자동 수집
●	수작업 보고 최소화
●	대시보드 상시 공개
 
7.	성공과 실패의 기준
언제 성공이고, 언제 접을 것인가?
7-1. 성공 기준
●	사전 정의한 KPI 달성
●	사용자 재사용률 유지
●	업무 담당자의 실질적 부담 감소
7-2. 실패 기준
●	사용률이 일정 수준 이하로 지속
●	수작업 보완 비용이 자동화 효과를 상회
●	법·제도 리스크가 상시 발생
7-3. 지속/중단 기준
●	기준 충족 시 확산
●	기준 미달 시 개선
●	개선 불가 시 중단
 
8.	마일스톤 설계
언제 무엇을 검증할 것인가?
8-1. 권장 마일스톤
●	M1: 문제·데이터 정의 완료
●	M2: 최소 기능(MVP) 구현
●	M3: 제한된 실사용 파일럿
●	M4: 확산 여부 결정
8-2. Pass 기준
●	실제 사용자 사용 여부
●	KPI 최소치 달성
●	운영 가능성 검증
8-3. Fail 결정 시점
●	두 번의 개선 후에도 지표 미달
●	비용 대비 효과 불충분
●	데이터·법적 한계 확인 시
 
9.	모델 독립성
특정 모델에 종속되지 않는가?
9-1. 구조 원칙
●	모델 교체 가능 구조
●	프롬프트·로직 분리
●	데이터와 모델 분리
9-2. 기술적 체크
●	API 추상화 계층 존재 여부
●	복수 모델 테스트 가능 여부
●	비용·성능 비교 가능성
9-3. 정책적 의미
●	특정 기업 종속 방지
●	기술 변화에 대한 유연성 확보
●	장기 운영 가능성
 
로그 시스템
1.	로그란 무엇인가
로그는
“이 시스템에서 언제, 누가, 무엇을 했고, 그 결과가 어땠는지”를 자동으로 남기는 기록 장치다.
중요한 점은
보고서를 쓰기 위한 기록이 아니라
사고를 막고, 문제를 발견하고, 성과를 증명하기 위한 운영 인프라라는 것이다.

2.	왜 반드시 갖춰야 하는가

로그가 없으면 다음 네 가지가 불가능하다.
1.	사고가 났을 때 원인을 알 수 없다
●	AI가 이상한 답을 했는지
●	데이터가 잘못 들어왔는지
●	사람이 잘못 쓴 것인지 구분 불가
2.	책임 소재를 구분할 수 없다
●	시스템 오류인지
●	사용자 오사용인지
●	정책·데이터 문제인지 설명 불가
3.	성과를 증명할 수 없다
●	“이 서비스가 정말 도움이 되었는가?”
●	“업무 시간이 줄었는가?”
→ 느낌만 있고 증거가 없다
4.	실패를 빨리 접을 수 없다
●	로그가 없으면 실패 여부를 판단할 기준 자체가 없다
즉,
로그는 “통제·감사”가 아니라
“안전한 실험과 빠른 판단”을 가능하게 하는 장치다.

3.	로그 시스템의 목적

공공 서비스 로그의 목적은 네 가지다.
1.	운영 안정성 확보
●	시스템이 정상적으로 돌고 있는지 상시 확인
2.	문제 원인 추적
●	장애, 오류, 민원 발생 시 정확한 원인 파악
3.	정책·행정 책임성 확보
●	“왜 이런 결과가 나왔는가”에 대한 설명 근거
4.	성과 측정과 의사결정
●	계속할지, 고칠지, 접을지 판단하는 근거 데이터

4.	최소 요건 1
반드시 남겨야 할 로그 항목

복잡할 필요 없다. 최소한 아래는 자동으로 남아야 한다.
1.	언제
●	날짜와 시간 (자동 기록)
2.	누가
●	시민 / 공무원 / 시스템
●	개인식별이 필요 없으면 내부 ID로 충분
3.	무엇을 요청했는가
●	버튼 클릭
●	민원 제출
●	AI 질의
●	문서 검토 요청 등
4.	시스템이 무엇을 했는가
●	처리 성공 / 실패
●	AI 응답 생성
●	규칙 적용
●	사람이 개입했는지 여부
5.	결과는 어땠는가
●	성공 / 오류
●	처리 시간
●	재시도 여부
이 다섯 가지만 있어도
“무슨 일이 벌어졌는지”는 재구성할 수 있다.

5.	최소 요건 2
로그는 반드시 한 곳에 모여야 한다

엑셀, 수기 보고, 부서별 파일은 로그가 아니다.
최소 요건은 다음이다.
●	모든 시스템 로그는 자동으로 중앙 저장소에 모인다
●	사람이 옮기지 않는다
●	수정할 수 없다 (읽기 전용)
왜냐하면
사고가 날 때 “각자 자료를 모으는 순간” 이미 실패했기 때문이다.

6.	최소 요건 3
사람이 읽을 수 있어야 한다

로그는 개발자만 보는 것이 아니다.
다음이 가능해야 한다.
●	오늘 사용자 수는 몇 명인가
●	AI 응답 오류는 얼마나 발생했는가
●	처리 시간이 늘어나는 추세인가
●	특정 부서·업무에서 실패가 집중되는가
이를 위해서는
●	표
●	간단한 그래프
●	빨간색 경고 표시
정도만 있어도 충분하다.
“대시보드 한 화면에서 상황이 보이는 것”
이것이 최소 요건이다.

7.	최소 요건 4
AI 서비스에 반드시 필요한 추가 로그

AI가 들어가면 로그가 더 중요해진다.
추가로 반드시 남겨야 할 것은 다음이다.
1.	AI가 사용한 정보의 출처
●	어떤 데이터
●	어떤 문서
●	어떤 규칙
2.	AI 응답에 사람이 개입했는지 여부
●	자동 처리
●	사람 검토 후 승인
●	사람 판단으로 수정
3.	오류 유형
●	데이터 부족
●	모호한 질문
●	정책 충돌
●	시스템 오류
이것이 없으면
“AI가 왜 그런 판단을 했는가”를 절대 설명할 수 없다.

8.	어떻게 구축해야 하는가 (원칙)

공무원이 기억해야 할 원칙은 세 가지다.
1.	자동으로 남겨라
●	사람이 입력하면 실패한다
2.	업무 흐름에서 자연스럽게 생기게 하라
●	버튼, 처리 단계마다 자동 기록
3.	처음부터 완벽하게 하려 하지 말라
●	최소 항목부터 시작
●	필요하면 늘린다

9.	어떻게 활용해야 하는가

로그는 쌓아두는 것이 아니라 써야 한다.
권장 활용 방식은 다음이다.
1.	주 1회 운영 점검
●	사용률
●	오류율
●	처리 시간
2.	마일스톤 판단 자료
●	다음 단계로 갈지
●	고칠지
●	접을지
3.	감사·국회·언론 대응
●	“감으로 설명”이 아니라
●	“데이터로 설명”

10.	로그 시스템 한 줄 요약
로그는
감시 도구가 아니라
공공 AI 서비스를 안전하게 실험하고, 실패를 빨리 인정하고, 성공을 증명하기 위한 최소한의 장치다.
 
로그 보고서
핵심은 “사람이 보고 판단해야 할 것”과 “시스템이 알아서 알려줘야 할 것”을 명확히 나누는 것이다.

1.	실시간 자동 알림 대상

실시간 알림의 목적은 보고가 아니라 즉각적인 대응이다.
따라서 “지금 바로 사람이 개입해야 하는 상황”만 포함한다.

1-1. 서비스 장애·중단 관련
즉시 알림 필요
●	서비스 응답 불가
●	특정 기능(민원 접수, AI 응답, 조회 등) 실패율 급증
●	처리 시간이 기준치를 초과한 상태가 일정 시간 지속
알림 이유
→ 시민 불편이 즉시 발생하며, 방치 시 신뢰 하락으로 직결

1-2. AI 응답 이상 징후
즉시 알림 필요
●	AI 응답 오류율 급증
●	“응답 불가 / 판단 불가” 비율 급상승
●	사람이 개입한 비율이 평소 대비 급증
알림 이유
→ 데이터 문제, 정책 변경 누락, 모델 이상 가능성

1-3. 데이터 이상
즉시 알림 필요
●	데이터 갱신 실패
●	특정 데이터 소스 미연결
●	데이터 양 급감 또는 급증
알림 이유
→ AI 판단 오류의 가장 흔한 원인

1-4. 보안·권한 이상
즉시 알림 필요
●	비정상적 접근 시도
●	권한 없는 기능 호출
●	동일 사용자 반복 오류 요청
알림 이유
→ 보안 사고 및 오남용 조기 차단

1-5. 정책·규칙 충돌
즉시 또는 준실시간 알림
●	규칙 간 충돌 발생
●	AI 판단과 규칙 판단 불일치 급증
알림 이유
→ 정책 업데이트 누락 가능성

2.	실시간 알림 설계 원칙

●	알림은 “적을수록 좋다”
●	한 알림 = 한 행동
●	알림에는 반드시 포함
○	무엇이 문제인가
○	언제부터 시작됐는가
○	어느 기능/업무인가
○	즉시 취할 수 있는 조치
알림이 잦아지면
→ 아무도 보지 않게 된다.

3.	주간 자동 보고서의 목적

주간 보고서는 “감시 보고서”가 아니라
의사결정 자료다.
●	계속할 것인가?
●	개선할 것인가?
●	중단할 것인가?
를 판단하기 위한 최소 정보만 담는다.

4.	주간 자동 보고서 필수 항목
4-1. 사용 현황
반드시 포함
●	주간 사용자 수
●	신규 사용자 수
●	반복 사용 비율
●	주요 기능별 사용량
의미
→ 실제로 쓰이고 있는가?
4-2. 처리 성과
반드시 포함
●	평균 처리 시간
●	처리 시간 변화 추세
●	자동 처리 비율
●	사람 개입 비율
의미
→ 업무 효율이 개선되고 있는가?
4-3. 품질 지표
반드시 포함
●	오류율
●	재처리 요청 비율
●	AI 응답 채택률
●	민원 재발생률
의미
→ “편하지만 틀린 서비스”는 아닌가?
4-4. 안정성 지표
반드시 포함
●	장애 발생 횟수
●	평균 복구 시간
●	데이터 갱신 성공률
의미
→ 운영 가능한 서비스인가?
4-5. 사용자 반응
가능하면 자동 수집
●	간단 만족도
●	불만 키워드 요약
●	자주 발생한 질문/오류 유형
의미
→ 현장의 체감은 어떤가?

5.	AI 서비스 전용 주간 보고 항목

AI가 들어간 경우 반드시 추가한다.
●	AI 요청 대비 성공 응답 비율
●	사람이 수정한 응답 비율
●	판단 불가 사유 TOP 항목
●	데이터 부족 원인 요약
이 항목은
“모델을 바꿀지, 데이터를 보완할지, 규칙을 강화할지”를 결정하는 근거다.

6.	주간 보고서 자동화 원칙

●	사람이 쓰지 않는다
●	정해진 형식으로 자동 생성
●	같은 포맷을 유지한다
●	전 주 대비 변화가 한눈에 보이게 한다
가장 중요한 질문은 이것이다.
“이 보고서를 보고 이번 주에 어떤 결정을 할 수 있는가?”

7.	실무자를 위한 한 장 요약

실시간 알림
→ 지금 당장 사람이 개입해야 하는 문제
주간 보고서
→ 다음 단계를 결정하기 위한 증거
로그는
쌓는 것이 목적이 아니라
알림과 판단으로 이어질 때 비로소 의미가 있다.

이유

3. 마일스톤별 Pass / Stop 기준표
이 표의 목적은 하나다.
“정치적 판단이 아니라, 기준으로 결정하게 만드는 것”

M1. 문제·데이터 정의
Pass
●	문제 정의가 사용자 인터뷰로 검증됨
●	핵심 데이터 소유·접근 가능
Stop
●	데이터 확보 불가
●	법·제도상 사용 불가

M2. MVP 구현
Pass
●	핵심 기능 실제 작동
●	로그 정상 수집
●	사용자 테스트 가능
Stop
●	로그로 상태 파악 불가
●	수작업 없이는 운영 불가

M3. 파일럿 운영
Pass
●	실제 사용자 사용
●	자동 처리 비율 증가
●	오류율 관리 가능 수준
Stop
●	사용률 미미
●	사람 개입 비용이 효과를 초과

M4. 확산 결정
Pass
●	KPI 지속 달성
●	담당 부서 재사용 의사 명확
●	타 부처 확장 가능성 확인
Stop
●	효과 대비 운영 비용 과다
●	정책·법적 리스크 상존

최종 정리
알림 임계값은
“지금 당장 개입해야 할 상황”만 잡아내기 위한 장치다.
주간 보고서는
“다음 결정을 내리기 위한 증거”다.
마일스톤 기준은
“잘 만든 서비스보다, 잘 접는 서비스를 가능하게 하는 규칙”이다.
이 세 가지가 함께 있어야
공공 AI 서비스가 실패해도 안전하고, 성공하면 확산될 수 있다.
 
참고자료 
영국 GDS Service Standard 한국어 요약
GDS Service Standard의 핵심 메시지는 단순하다.
“공공 서비스는
아이디어가 좋아서가 아니라,
사용자 문제를 실제로 해결하고 있다는 증거가 있을 때만
존재할 수 있다.”
이를 위해 14가지 기준을 요구한다.

1.	사용자를 이해하라

●	가정하지 말고 실제 사용자를 조사하라
●	시민, 현업 공무원, 이해관계자를 직접 만나야 한다
●	문서가 아니라 행동과 불편을 관찰하라
의미
→ “우리가 필요하다고 생각한 것”은 기준이 아니다

2.	문제 전체를 해결하라

●	한 부처, 한 시스템, 한 단계만 보지 말라
●	사용자가 겪는 문제의 처음부터 끝까지를 보라
의미
→ 디지털화된 불편을 만들지 말라는 경고

3.	여러 채널에서 일관된 서비스를 제공하라

●	온라인, 오프라인, 전화, 창구가 서로 충돌하면 안 된다
●	어느 채널을 써도 같은 결과로 이어져야 한다
의미
→ 디지털 서비스는 기존 행정을 보완해야지 분절해서는 안 된다

4.	누구나 쓸 수 있도록 설계하라

●	장애인, 고령자, 디지털 약자를 기본 사용자로 가정
●	접근성은 선택이 아니라 필수
의미
→ “대부분 쓸 수 있다”는 공공 서비스 기준이 아니다

5.	단순하게 만들어라

●	절차를 줄이고
●	용어를 바꾸고
●	선택지를 최소화하라
의미
→ 복잡함은 기술 문제가 아니라 설계 실패다

6.	빠르게 만들고, 자주 테스트하라

●	완벽한 계획보다 빠른 실험
●	작은 기능부터 실제 사용자에게 검증
의미
→ 실패를 늦추는 것이 아니라, 실패를 빨리 발견하라

7.	보안과 개인정보 보호를 기본으로 설계하라

●	나중에 붙이지 말고 처음부터 포함
●	최소 수집, 최소 보관 원칙
의미
→ 신뢰를 잃으면 서비스는 끝난다

8.	데이터를 기반으로 결정하라

●	성공 기준을 미리 정의
●	사용률, 처리 시간, 오류율을 지속 측정
의미
→ “잘 되고 있다”는 느낌은 근거가 아니다

9.	기술이 아니라 사용자 요구에 맞춰 설계하라

●	최신 기술 사용은 목적이 아니다
●	기술은 언제든 바꿀 수 있어야 한다
의미
→ 기술 종속은 공공 서비스의 실패다

10.	개방 표준과 재사용을 우선하라

●	다른 서비스가 재사용할 수 있도록 설계
●	특정 공급자에 묶이지 말 것
의미
→ 세금으로 만든 것은 다시 쓸 수 있어야 한다

11.	지속적으로 개선하라

●	출시가 끝이 아니다
●	서비스는 살아있는 운영 대상이다
의미
→ ‘완성’이라는 개념은 없다

12.	팀에 필요한 역량을 갖춰라

●	정책, 현업, 디자인, 기술이 함께 일해야 한다
●	외주 관리만으로는 기준을 충족할 수 없다
의미
→ 공공 디지털 서비스는 조직 역량 문제다

13.	운영 가능성을 증명하라

●	장애 대응
●	비용
●	유지보수
●	인수인계 계획이 있어야 한다
의미
→ “만들 수 있다”보다 “계속 돌릴 수 있다”가 중요하다

14.	공개적으로 검증받아라

●	외부 평가(Service Assessment)를 통과해야 한다
●	기준 미달 시 출시 불가
의미
→ 정치적 결정이 아니라 기준으로 판단
한 줄 요약
GDS Service Standard는 “공공 서비스는 선의로 만들 수 없고, 증거로만 존재할 수 있다”는 규칙이다.
